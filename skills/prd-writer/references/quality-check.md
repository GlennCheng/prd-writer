# Quality Check & Contradiction Detection

## 執行時機

Step 3 產出完整 PRD 後執行。結果以**提示方式呈現，不強制修正**。  
PM 選擇要修正哪些，AI 更新後重新驗證，直到 PM 確認或略過。

---

## 品質驗證項目

### 1. 完整性
對照 prd-template.md 的 mandatory 區塊：
- Story Background
- User Scenarios & Testing（含 Acceptance Scenarios）
- Scope
- Requirements（至少 3 個 FR）
- Success Criteria（至少 2 個 SC）

❌ 若有 mandatory 區塊為空或只有佔位文字 → 列出缺少的區塊

### 2. 清晰度
掃描全文，找出未量化的模糊形容詞：

| 模糊詞 | 建議轉譯 |
|-------|---------|
| 快速、即時 | 具體時間（「2 秒內」） |
| 友善、直覺 | 具體行為（「新使用者不需閱讀說明即可完成」） |
| 大量、許多 | 具體數字（「超過 1,000 筆」） |
| 定期 | 具體頻率（「每天一次」） |

### 3. 可測性
檢查每個 FR 和 SC：
- 能否設計一個測試案例來驗證它？
- 是否有明確的成功條件？
- 「System MUST X」中的 X 是否可被客觀觀察？

❌ 無法客觀驗證的需求 → 建議加上量化條件或驗證方法

### 4. PM 語言
掃描技術術語：

常見違規詞（建議轉譯為使用者語言）：
- API、endpoint → 「系統連接」
- 資料庫、schema → 「資料儲存方式」
- 前端/後端、框架 → 不提，只描述行為
- 快取 → 「暫存」或描述效果
- 非同步、同步 → 「即時」或「稍後更新」

### 5. `[NEEDS CLARIFICATION]` 數量
❌ 超過 3 個 → 建議優先處理最關鍵的，其餘標記為 Assumption

## 矛盾偵測

### 矛盾類型

| 類型 | 範例 |
|------|------|
| **範圍矛盾** | Scope 限定「僅 Desktop」，但 FR-003 描述了 Mobile 行為 |
| **需求衝突** | FR-001 要求「即時更新」，但 Experience 寫「可接受延遲」 |
| **優先級矛盾** | P1 User Story 依賴 P3 功能才能運作 |
| **指標矛盾** | SC-001 期望「2 秒內載入」，但 FR 要求「載入所有子項目資料」 |
| **假設衝突** | Assumptions 假設「既有登入不變」，但 FR 新增了角色權限 |

### 矛盾提示格式

```
⚠️ 偵測到 N 個潛在矛盾：

矛盾 1：[類型]
  • 區塊 A 說：「...」
  • 區塊 B 說：「...」
  → 建議：[AI 推薦的解決方式]

矛盾 2：...

要修正哪些？（回覆編號，或說「略過」繼續）
```

## 品質報告格式

```
📋 PRD 品質報告

✅ 通過：完整性、可測性
⚠️ 建議改善：
  - 清晰度：FR-002 中「快速回應」建議量化為具體秒數
  - PM 語言：SC-003 中出現「API 呼叫次數」，建議改為使用者語言
  - 矛盾：Scope 與 FR-005 範圍不一致

要修正哪些？（回覆項目名稱，或說「略過全部」）
```
